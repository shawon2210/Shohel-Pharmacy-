## P1 — Cross-module event/trigger mechanism

**Problem:** Modules can't communicate directly. Agent is the only glue — if it forgets a step (e.g. update Sheets after sending an email), data becomes inconsistent.

**Desired behavior:** Module A emits an event (e.g. `gmail.draft_sent`) → Module B reacts (e.g. `sheets.update_cell` on the contact's "last_contact" column).

**Scope:** Event bus or webhook system within the sandbox runtime. Could be SQLite-backed event log + polling, or in-process pub/sub.

## P1 — Twitter thread support

**Problem:** Twitter module only supports single tweets (280 chars). Threads are the highest-engagement format on X — long-form ideas, storytelling, tutorials.

**Desired behavior:**
- `twitter_create_thread` tool accepting array of tweet contents
- Thread stored as linked posts in SQLite
- Published sequentially via Twitter API v2 (each tweet replies to the previous)
- UI shows thread as connected cards

**Scope:** New MCP tool, DB schema extension (parent_id or thread_id), publisher logic for chained replies.

## P1 — Image/media upload for social modules

**Problem:** All social modules are text-only. Pure text content has significantly lower engagement than media content on every platform.

**Desired behavior:**
- Agent can reference workspace files (images generated by image agent) as attachments
- Twitter: upload media → attach to tweet
- LinkedIn: image posts
- Reddit: link/image post types

**Scope:** Media upload API integration per platform, file reference system from workspace, new tool parameters.

## P2 — GitHub write capabilities

**Problem:** GitHub module is read-only. Can't create issues, post PR review comments, trigger workflows. DevRel and project management scenarios are incomplete.

**Desired tools:**
- `github_create_issue` — title, body, labels, assignees
- `github_comment_on_issue` / `github_comment_on_pr`
- `github_create_release` — tag, name, body, draft flag

**Scope:** New MCP tools, GitHub API v3 POST/PATCH calls via integration broker.

## P2 — Post-publish engagement tracking

**Problem:** After publishing, there's no feedback loop. Agent can't know what content performs well, can't suggest strategy adjustments, can't respond to comments.

**Desired behavior:**
- Periodic polling of published post metrics (likes, shares, comments, impressions)
- Stored in local SQLite, queryable via MCP tool (`twitter_get_post_analytics`)
- Agent can ask "which of my recent posts performed best?" and get data

**Scope:** New MCP tools per platform, background polling worker, analytics DB table.

## P2 — Content template/signature system

**Problem:** Agent writes every piece of content from scratch. No brand voice persistence, no email signatures, no reusable templates.

**Desired behavior:**
- Workspace-level templates stored in YAML/markdown (e.g. `templates/email-signature.md`, `templates/twitter-promo.md`)
- Agent can reference templates when creating content
- Templates include variables ({{name}}, {{company}})

**Scope:** Template storage in workspace dir, MCP tool to list/read templates, agent prompt integration.

## P3 — LinkedIn carousel support

Rich carousel posts are LinkedIn's highest-engagement format. Requires PDF generation from slides + LinkedIn document upload API.

## P3 — Reddit flair/crosspost

Many subreddits require post flair. Crossposting to multiple subreddits is a common distribution pattern. Requires Reddit API flair endpoints + crosspost API.

## P3 — Sheets real-time sync

Currently snapshot-based. Agent reads sheet data at call time but can't detect external changes. A lightweight polling mechanism or change detection would enable reactive workflows.
